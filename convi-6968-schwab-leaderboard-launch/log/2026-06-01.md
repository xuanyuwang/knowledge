## Details

### Agent Tab

- The current Agent leaderboard uses `useGetQAStats(...)`, which builds a `RetrieveQAScoreStats` request.
- The shared QA filter mapper includes users, teams, groups, conversation duration buckets, scorecard template, criterion identifiers, use case, score ranges, exclude deactivated users, scorecard statuses, and moment groups.
- `RetrieveQAScoreStats` also carries date range, `conversationTimeRangeField`, `scoreResource`, and `filterToAgentsOnly`.
- This makes `RetrieveQAScoreStats` the closest match for current leaderboard QA filter parity.
- The blocker is the response shape: it exposes aggregate scorecard counts, but no template dimension.
- `RetrieveQAConversations` can be transformed into scorecard-level records with `scorecardTemplateId`, so FE can derive counts grouped by template per agent.
- `RetrieveQAConversations` does not support `filterToAgentsOnly`, which is the main parity gap versus the existing Agent leaderboard path.

### Manager Tab

- The existing `Scorecards completed` column uses `RetrieveScorecardStats` via `useScorecardStats(...)`.
- The manager page first resolves managers from the selected child teams using manager-like roles, then builds a request grouped by `ATTRIBUTE_TYPE_AGENT`.
- The request includes the selected date range and `conversationDurationBuckets`, but not submitter, template, or reviewer-specific scorecard filters.
- The table row uses `averageScorecardCompletedPerUser` as the displayed value for `Scorecards completed`.
- `RetrieveScorecardStats` does not provide any template breakdown, so it cannot directly answer grouped-by-template manager requirements.

### Alternative Data Source

- `ListScorecards` is the clearest explicit API for "scorecards submitted by manager grouped by template" because it supports:
- `submitterUserNames`
- `creatorUserNames`
- `templateName`
- `startSubmitTime` / `endSubmitTime`
- `agentUserNames`
- `groupNames`
- `conversationNames`
- `scorecardNames`
- The main downside is filter parity. `ListScorecards` does not naturally cover all leaderboard filters such as QA-specific conversation bucket and voicemail-style filtering without additional prefiltering or backend support.
