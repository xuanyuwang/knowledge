Added a clear design guideline at the top of the investigation document:

**Principle**: A virtual group is just a shortcut to a group of individual users. The behavior should be identical to processing those individual users directly.

**Current Problem**: Code correctly expands virtual groups to users for filtering, but then incorrectly aggregates stats under users' TEAM groups instead of the requested virtual group.

## Bug Flow Analysis

Created detailed trace document: `.tmp/bug-flow-analysis.md`

### Exact Bug Locations

**Bug #1: Virtual groups filtered out**
- File: `insights-server/internal/shared/common.go`
- Line 805-806: Only TEAM groups added to `filterTeamGroupNames`
- Line 854-856: Virtual group memberships skipped

**Bug #2: Wrong group mapping for aggregation**
- File: `insights-server/internal/analyticsimpl/retrieve_qa_score_stats.go`
- Line 398-403: Uses `userNameToGroupNamesMap` which contains TEAM groups, not the filtered virtual group

### Key Finding

The code has **two distinct problems**:

1. **ListUsersMappedToGroups** excludes virtual groups from `groupsToAggregate` and `userNameToGroupNamesMap`
   - This makes virtual groups disappear from results

2. **userNameToGroupNamesMap contains the wrong groups**
   - Current: Maps users to ALL their TEAM groups
   - Should: Map users to the groups specified in the FILTER

Example with filter `VG1` containing `[U1, U2, U3]`:
- Current map: `{"U1": ["TeamA"], "U2": ["TeamB"], "U3": ["TeamC"]}`
- Correct map: `{"U1": ["VG1"], "U2": ["VG1"], "U3": ["VG1"]}`

This explains why results show TeamA/TeamB/TeamC instead of VG1!

## CRITICAL DISCOVERY: Current Behavior is INTENTIONAL

### Existing Test Found

**Test**: `B_GS_2_TeamAndDynamicGroupTypesHandledSeparately`
**File**: `common_user_filter_test.go:1720-1780`

**Test Documentation**:
```
TEAM and DYNAMIC groups are both passed through to buildUserGroupMappings,
but only TEAM groups appear in the user-to-group mappings and GroupsToAggregate.
DYNAMIC groups are used only for filtering (selecting members) but excluded
from the output mappings.
```

**Test Assertions**:
- ✅ Dynamic group should NOT appear in user-to-group mapping
- ✅ TEAM groups SHOULD appear in mappings for filtered users

### What This Means

The current behavior is **BY DESIGN**, not a bug:

1. **Virtual groups** → Used for FILTERING (selecting which users)
2. **TEAM groups** → Used for AGGREGATION (grouping results)

**Example Flow**:
- Filter by virtual group `VG1` containing [User1, User2, User3]
- User1 is member of TeamA, User2 is member of TeamB, User3 is member of TeamC
- **Current Result**: Stats grouped by TeamA, TeamB, TeamC
- **Expected per test**: This is CORRECT behavior

This explains the manual testing results where TEAM groups appeared instead of the virtual group!
