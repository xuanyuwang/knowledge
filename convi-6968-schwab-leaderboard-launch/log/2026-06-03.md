# Project Log - 2026-06-03

## Progress

- Revisited the Manager tab API decision after the backend change to `RetrieveScorecardStats`.
- Confirmed the current `RetrieveScorecardStats` ClickHouse path rewrites `scorecard_time` to `scorecard_submit_time`.
- Confirmed the current `RetrieveScorecardStats` ClickHouse path also rewrites `agent_user_id` to `submitter_user_id`, so the Manager `Scorecards completed` aggregate is now submitter-attributed.
- Updated the Manager drawer recommendation from `ListScorecards.creatorUserNames` to `ListScorecards.submitterUserNames`.
- Documented that `RetrieveQAConversations` is not a drop-in replacement for the Manager drawer because it does not currently filter by `submitter_user_id`, does not apply `scorecardReviewerAudience` in the traced ClickHouse path, and does not use `scorecard_submit_time` as its default time basis.
- Documented that `RetrieveQAScoreStats` is not a drop-in replacement for `RetrieveScorecardStats` because its `QA_ATTRIBUTE_TYPE_AGENT` grouping is still based on `agent_user_id`, not `submitter_user_id`.

## Decision Update

The Manager `Scorecards completed` metric should now be described as:

`Distinct submitted scorecards, attributed to the scorecard submitter.`

Current recommended split:

- Main Manager table: keep `RetrieveScorecardStats`.
- Manager side drawer MVP: use `ListScorecards` with `submitterUserNames`, `startSubmitTime`, `endSubmitTime`, and `scorecardView = FULL`, then group by template in FE.
- Future Manager side drawer: prefer `RetrieveQAConversations` only after backend support exists for submitter filtering and scorecard submit-time filtering.
- Do not replace `RetrieveScorecardStats` with `RetrieveQAScoreStats` unless `RetrieveQAScoreStats` gains equivalent submitter attribution and submit-time behavior.
