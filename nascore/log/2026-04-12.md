**Validation Fix for N/A Score Support**
- Created analysis document: `/director/.tmp/VALIDATION_FIX_ANALYSIS.md`
- **Problem:** validation.ts requires all scores to be numbers >= 0, fails for N/A options with `score: null`
- **Fix approach:** Pass `options` array to validation functions, check `isNA` flag to allow null scores
- **Why no other approach:** 
  - Can't identify N/A scores from scores array alone (no isNA flag in SettingsScore type)
  - Allowing null universally would be too permissive (normal options should require scores)
  - No type changes needed - uses existing data structures
- **Changes made:**

  - Updated `validateNumOccurrencesScoreTypeConfigurationItem` (lines 284-294)

  - For normal options: require `score: number >= 0`

**CRITICAL BUG FIXED: Score Initialization for New Criteria**
- Created detailed report: `/director/.tmp/NA_SCORE_VALIDATION_BUG_REPORT.md`
- **Bug:** "At least one score is not configured properly" validation error when creating new criteria
- **Malformed data:** `[{"score":1},{"score":2},{"value":2,"score":null}]` - missing "value" field
- **Root causes:**
  1. defaultCriterion pollution: N/A option copied from previous criterion
  2. React Hook Form useFieldArray creates empty objects `[{},{}]` before initialization
  3. Timing issue: useOnMount runs too late
- **Solution:** Initialize scores in handleAddCriterion BEFORE adding to form
  ```typescript
  // Filter out N/A from defaultCriterion
  const optionsWithoutNA = defaultCriterion.settings.options.filter(opt => !opt.isNA);
  
  // Normalize options to sequential indexes
  newCriterion.settings.options = optionsWithoutNA.map((opt, i) => ({
    label: opt.label,
    value: i  // 0, 1, 2...
  }));
  
  // Initialize scores
  newCriterion.settings.scores = optionsWithoutNA.map((opt, i) => ({
    value: i,
    score: opt.value  // Original value WAS the score
  }));
  ```
- **Result:** Scores properly initialized: `[{"value":0,"score":1},{"value":1,"score":0}]`
- **Why this fix is best:**
  - Prevents problem at source (before React Hook Form touches it)
  - Maintains defaultCriterion feature (copies options, filters N/A)
  - Clear separation of concerns
  - Minimal code changes
- **Files changed:**
  - TemplateBuilderFormConfigurationStep.tsx: Score initialization in handleAddCriterion
  - CriteriaLabeledOptions.tsx: Simplified useOnMount, removed logging
- **Testing:**
  - [x] Create new criterion with N/A ŌåÆ validates successfully
  - [x] Create criterion, edit, create another ŌåÆ N/A not copied
  - [x] Save/publish template with N/A null scores ŌåÆ succeeds

**Investigation: Why Main Branch Works Without Score Initialization**
- **Question:** Main branch doesn't initialize scores in handleAddCriterion, but still works. Why?
- **Answer:** React Hook Form behavior + useOnMount timing
  - Main branch: `newCriterion.settings.scores` not set ŌåÆ `undefined` in form
  - useWatch returns `undefined` ŌåÆ `!watchedScoresField?.length = true`
  - CriteriaLabeledOptions.useOnMount sees condition and initializes scores
- **Feature branch:** Scores initialized in handleAddCriterion ŌåÆ `watchedScoresField = [...]`
  - useOnMount sees scores exist ŌåÆ skips initialization (correct!)
- **Conclusion:** Both approaches work, feature branch approach is better for:
  1. Prevents React Hook Form from creating empty placeholders `[{},{}]`
  2. Allows N/A migration to work correctly
  3. Separation of concerns: handleAddCriterion sets up base criterion, CriteriaLabeledOptions handles feature enhancements
