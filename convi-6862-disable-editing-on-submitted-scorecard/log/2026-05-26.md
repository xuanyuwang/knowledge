### Template Audience Runtime Resolution Investigation

- The `director.scorecard_templates` table stores raw audience config in `audience` JSONB.
- The backend does not persist a resolved member list for template audience.
- `resolved_audience` is output-only in proto and is assembled on demand in template read/transform paths.
- `GetScorecardTemplate` and related template-to-proto flows call runtime resolution helpers that:
  - parse the stored `Audience` JSON
  - collect referenced user IDs and group IDs
  - fetch current user/group resources from user service
  - build `ScorecardTemplate.ResolvedTemplateAudience`
- AutoQA template applicability also resolves dynamically:
  - direct user audience uses current user identity
  - group audience uses current group membership lookup
  - team audience uses the agent's current `TeamID`
- This means changes to team/group membership are reflected by current runtime reads and applicability checks rather than waiting for a template-side recompute.

### Current `teams` Handling Gaps

- Display resolution treats `teams` as group-backed resources:
  - team resource names are converted to group IDs
  - current group objects are fetched
  - those groups are adapted back into `user.Team` objects for `resolved_audience`
- AutoQA applicability checks `teams` against the agent's current `TeamID`, so that path is also runtime-based.
- But not every audience consumer handles `teams` consistently:
  - `ListCurrentScorecardTemplates.resolveUsersToTemplateAudiences` only matches `users` and `groups`
  - `shared/qa.ResolveUsersFromAudience` also only resolves `users` and `groups`
  - as written, a team-only audience can behave incorrectly in those paths
