### Removed BooleanFilter chip from filter bar

The "Agents only" filter was rendering in two places:
1. Inside the "+Filters" dropdown (via `useListAgentOnlyLevelSelect` level select hook) — **keep**
2. As a standalone BooleanFilter chip appended to the filter bar (via `components` map entry) — **removed**

Removed `[FilterKey.LIST_AGENT_ONLY]` from the `OrderedFilterBarFilters` components map in both `usePerformanceFilters.tsx` and `useLeaderboardsFilters.tsx`. `OrderedFilterBar` already returns `null` for filter keys without a component (line 226), so the filter bar gracefully skips it. The filter state, persistence, "+Filters" menu entry, and API pass-through are all unchanged.

Updated implementation plan: Phase 3.2 and 4.2 (register BooleanFilter component) marked as removed.

### Phase 1.2: BE handler wiring

Investigated all 16 analytics API handlers in `go-servers/insights-server/internal/analyticsimpl/`. Found:
- 11 handlers hardcode `listAgentOnly := true`
- 1 handler (LiveAssistStats) already reads `req.GetFilterToAgentsOnly()`
- 4 handlers use old pattern or are manager-only (no change needed)

Updated 11 handlers to read from request: `listAgentOnly := req.GetFilterToAgentsOnly()`. FE always sends the field (defaults to `true`), so effective behavior is preserved for new FE. Old FE clients without the field get `false` (include managers) — acceptable behavior change.

No proto change needed — kept `bool filter_to_agents_only` as-is.

Doc: `phase-1.2-be-handler-wiring.md`

### FE default: listAgentOnly true when flag is off

Changed `modifiedFiltersState` in both `usePerformanceFilters.tsx` and `useLeaderboardsFilters.tsx` to set `listAgentOnly: true` (was `undefined`) when `enableAgentOnlyFilter` feature flag is off. This ensures the FE always sends `filter_to_agents_only=true` to the BE, preserving current behavior when the BE switches from hardcoded `true` to reading from the request.

FE PR: [director #17356](https://github.com/cresta/director/pull/17356)
