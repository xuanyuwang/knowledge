# CONVI-6808: Frontend N/A Analysis

## Screenshot Observation

The most recent weekly column shows **N/A for ALL criteria** (including non-metadata ones like "Explain Service Frequency", "Explain Billing Frequency", etc.) while previous weeks show data normally.

## Frontend N/A Rendering Path

### How N/A is displayed

1. **Column headers** are generated by `createDateRangesInRangeForFrequency()` (`director-utils/src/datetime/dateUtils.ts:69`), creating weekly date ranges from `submitDateRange.startDate` to `submitDateRange.endDate`
2. Each column key is `startDate.format('YYYY-MM-DD')` (line 288 of `useColumnsForPerformanceProgression.tsx`)
3. **Data is populated** by `createColumnsForCriterion()` (`performance-progression/utils.ts:59`), keyed by `formatDayjs(s.groupedBy?.intervalStart)`
4. **N/A appears** when `row.columns[startDateFormatted]` returns `undefined` (line 293 of `useColumnsForPerformanceProgression.tsx`)

### Three conditions that produce N/A

| Condition | Location | When it fires |
|-----------|----------|---------------|
| No API data for the period | `utils.ts:67-70` | Backend didn't return a `QAScore` entry for that criterion+week |
| `weightSum === 0` | `utils.ts:72-74` | Backend returned data but all scores had `percentage_value < 0` |
| Date key mismatch | `useColumnsForPerformanceProgression.tsx:293` | Column key ≠ data key (formatting or timezone difference) |

## PI Architecture: Dual API Calls

The Performance Progression view makes **two separate `RetrieveQAScoreStats` requests** (`PerformanceProgression.tsx`):

| Call | GroupBy | Used For | Options |
|------|---------|----------|---------|
| Per-criterion (line 120) | `[CRITERION, TIME_RANGE]` | Individual criterion rows + drives chapter column population | `criterionFilter`, no `enableAutofailScoring` |
| Whole-template (line 156) | `[TIME_RANGE]` | "All criteria" aggregate stats, chapter averages | `enableAutofailScoring: true` |

Both use the **same `filtersState.submitDateRange`** → identical `filterByTimeRange`. The date range cannot differ between calls.

**Critical dependency**: `createColumnsForChapter()` (line 133-170 of utils.ts) only iterates dates present in CHILDREN rows (`allDates`), not all `chapterStats` dates. If the per-criterion response has no data for a week, ALL rows (chapters included) show N/A — even if the whole-template response has data.

### Key Backend Code Path Difference

`retrieve_qa_score_stats.go:203-219`: When groupBy does NOT include CRITERION and `criterionIdentifiers` is empty, the backend auto-populates with scoreable criteria. When CRITERION IS in groupBy, this auto-filter is skipped. This shouldn't reduce results (it means per-criterion gets ALL criteria).

### Most Likely Explanation for Original Screenshot

1. **Data processing lag** (most likely) — The screenshot was captured around May 7-8 when the data pipeline may not have finished populating `conversation_d` for recent conversations. The PI query JOINs `score_d` with `conversation_d`; if `conversation_d` lags behind, the JOIN produces zero results for the most recent period. It has since caught up.

2. **Backend response caching** — A stale empty response was cached (`retrieve_qa_score_stats.go:136`) for the per-criterion request and has since expired.

3. **Voicemail exclusion filter interaction** — The per-criterion request includes a voicemail exclusion filter (`momentGroups` with `excludedMoments`), which routes through `qaScoreStatsClickhouseQueryWithMetadataView` instead of `qaScoreStatsClickhouseQuery`. A transient issue in the metadata view JOIN could have temporarily excluded conversations.
