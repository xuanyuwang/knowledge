## Progress

Fixed two AutoQA bugs in CriteriaLabeledOptions and TemplateBuilderAutoQA:

**Fix 1: N/A label in AutoQA dropdowns**
- Bug: N/A option showed "N/A (2)" instead of "N/A (no score)" when score is null
- Root cause: `label: ${option.label} (${score ?? option.value})` at TemplateBuilderAutoQA.tsx:242
- Fix: Check for `option.isNA && score == null` to show "no score" instead of falling back to option.value
- Commit: dd96747dc4

**Fix 2: Score auto-population when disabling AutoQA**
- Bug: When unchecking "Automated scoring" on new criteria, scores auto-populate as 0, 1, ... instead of staying empty
- Root cause: Legacy scores migration at CriteriaLabeledOptions.tsx:94-96 was running inappropriately
- Fix: Remove legacy scores array migration, keep only N/A option migration (which simulates "Allow N/A" checkbox for old templates)
- Commit: 99aaed9705

**CRITICAL ISSUE FOUND: DEFAULT_CRITERION Incompatible with Decoupled Scoring**
- Created issue document: `/director/.tmp/DEFAULT_CRITERION_ISSUE.md`
- **Problem:** DEFAULT_CRITERION provides options but NO scores array
- **Impact:** After removing legacy migration, new criteria have broken score inputs
- **Root cause:** CriteriaLabeledOptions expects both options and scores to exist or both to be empty
- **Recommended fix:** Change DEFAULT_CRITERION to have empty options and scores arrays
  ```tsx
  settings: {
    options: [],  // ← Empty, let CriteriaLabeledOptions create them
    scores: [],
    showNA: true,
  }
  ```
- **Effect:** First option created via onAddLabel() with proper value: 0, score: 0
- **Priority:** CRITICAL - must fix before merging enableNAScore feature
- **Status:** DEFERRED - user noted this has always been the case with no bug reports, investigating further

**Validation Fix for N/A Score Support**
- Created analysis document: `/director/.tmp/VALIDATION_FIX_ANALYSIS.md`
- **Problem:** validation.ts requires all scores to be numbers >= 0, fails for N/A options with `score: null`
- **Fix approach:** Pass `options` array to validation functions, check `isNA` flag to allow null scores
- **Why no other approach:** 
  - Can't identify N/A scores from scores array alone (no isNA flag in SettingsScore type)
  - Allowing null universally would be too permissive (normal options should require scores)
  - No type changes needed - uses existing data structures
- **Changes made:**
  - Updated `validateSettingsScores` signature: added `options` parameter
  - Updated two call sites (lines 90, 486) to pass `item.settings.options`
  - Updated `validateNumOccurrencesScoreTypeConfigurationItem` (lines 284-294)
  - For N/A options: allow `score: null` or `number >= 0`
  - For normal options: require `score: number >= 0`
  - At least one non-zero score rule: N/A with null doesn't count, N/A with number > 0 does count
