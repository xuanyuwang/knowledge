## Details

### Root Cause Confirmed

The scorecard panel renders `CriterionInputDisplay` (via `CriterionMain.tsx:151` → `CriterionInput`) in **read-only** mode. The chain:

1. Form init (`utils.ts:712`): `String(score.numericValue)` → field value `["0"]`
2. Decoupled options (`CriterionInputDisplay.tsx:243-250`): `value: String(index)` — uses array index
3. Read-only display (`CriterionInputDisplay.tsx:262-270`): finds option by matching `fieldValue` against `option.value` (which is the index)
4. Field value `"0"` matches index 0 → wrong label when `option.value ≠ index`

### Template Configuration

predicted-resolution options are in reversed order from their values:
- `settings.options[0]`: value=1, label="Not Resolved"
- `settings.options[1]`: value=0, label="Resolved"

This means `option.value ≠ array index`, triggering the bug.

### Key Insight: Two Different Display Components

- `useScoreLabelProvider` (line 703-711): maps `option.value → label` — **CORRECT** for auto-scored values
- `CriterionInputDisplay` (line 243-250): maps `array index → label` in decoupled mode — **WRONG** for auto-scored values

The scorecard panel uses `CriterionInputDisplay`, not `useScoreLabelProvider`, for the main criterion display.

### Recommended Fix

Option B: Fix form initialization (`utils.ts:710-712`) to convert `numericValue` (which is `option.value`) to array index for decoupled scoring criteria. This fixes both read-only and interactive displays without changing DB schema or auto-scoring.
