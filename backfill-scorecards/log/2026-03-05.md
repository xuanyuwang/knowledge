## Details

### Why a script instead of the Temporal workflow (PR #25916)

PR #25916 adds a proper `ReindexScorecardsWorkflow` but isn't merged yet. For a quick one-time backfill of ~thousands of process scorecards, a direct PG→CH Python script is faster to execute.

### Key design decisions

- **Data source**: Uses `historic.scorecard_scores` (pre-computed percentage_value, weight, float_weight, max_value) instead of recomputing from `director.scores` + template JSON. Avoids reimplementing `ComputeCriterionPercentageScore` logic in Python.
- **scorecard_time difference**: CH `scorecard` table uses `process_interaction_at`, CH `score` table uses `created_at` (matches Go `buildScorecardRows` vs `buildScoreRows` behavior).
- **Distributed tables**: Writes to `scorecard_d` / `score_d` (distributed) so CH handles shard routing. No need for `findShardForProcessScorecard` logic.
- **Idempotent**: CH `ReplacingMergeTree` deduplicates on primary key, so re-running is safe.
- **is_dev_user**: Looked up from `app.users` per batch per customer, defaults to false if not found.

### Code review (2026-03-05)

- **Security**: Removed hardcoded CH password; script now requires `CH_PASSWORD` (env or `--ch-password`) for backfill/verify. CH host/port/user/database remain configurable via env.
- **Correctness**: Fixed `process_batch` to fetch `is_dev_user` per customer. Previously used `scorecards[0]["customer"]` for the whole batch, so batches spanning multiple customers would have wrong `is_dev_user` for agents outside the first customer. Spirit-only runs were unaffected.
- **Bug fix**: Replaced JOIN with EXISTS subquery for `scorecard_templates` — the table has composite PK including `revision`, causing 6x duplicate counts with JOIN.
- **Bug fix**: Users table is `app.users`, not `director.users`.
