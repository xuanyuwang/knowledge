## Details

- The Agent tab needs two distinct data jobs:
- aggregate submitted scorecard counts for all agents at once
- detailed submitted scorecards for a single agent, grouped by template, only when the drawer is opened
- A separate `RetrieveQAScoreStats` query remains the best fit for the Agent aggregate table because it matches the current QA score semantics and filter surface while allowing `scorecardStatuses = [MANUALLY_SUBMITTED]`.
- `RetrieveQAConversations` remains the best fit for the Agent on-demand drawer because it exposes QA scorecard detail rows that FE can group by template; it should also hardcode `scorecardStatuses = [MANUALLY_SUBMITTED]`.
- `RetrieveScorecardStats` and `ListScorecards` can each solve parts of the problem, but they are less aligned with the current Agent-tab semantics and filter parity.
- For Manager tab consistency, the main table should stay on `RetrieveScorecardStats`.
- For Manager drawer consistency, use `ListScorecards` for MVP for a single manager with `creatorUserNames`, `startSubmitTime`, and `endSubmitTime`, then group by template on FE.
- The Manager drawer data fetching should return a normalized provider shape so a future implementation can switch to `RetrieveQAConversations` after scorecard creator filtering is added.
- The current Manager `Scorecards completed` metric means distinct submitted scorecards attributed to `creator_user_id`, not generic reviewed scorecards and not necessarily `submitter_user_id`.
- `RetrieveQAConversations` can filter submitted scorecards via `scorecard_statuses = MANUALLY_SUBMITTED`, which becomes `scorecard_submit_time <> 0`.
- `RetrieveQAConversations` does not apply `scorecard_reviewer_audience` in the traced ClickHouse path, does not filter by `creator_user_id` or `submitter_user_id`, and does not use `scorecard_submit_time` as the default time range basis.
- For Agent FE, `QAScore.totalScorecardCount` is available in `RetrieveQAScoreStats`, but the existing Performance score response should not be reused because an empty status filter includes all matching statuses. Use a separate submitted-only request.
- For Manager FE, `numOfScorecardsCompleted` is already populated from `RetrieveScorecardStats`; the UI should keep that metric and only add click behavior.
- The drawer should not reuse `QAConversationExamplesDrawer` directly because that drawer assumes a selected template and renders conversation examples, while this feature needs all templates grouped into collapsible sections.
- The best matching UI patterns are Coaching Hub's parent-owned side drawer state and Coaching Report's `FullDrawer` plus `Accordion` structure.
