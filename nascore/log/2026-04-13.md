## Refactoring: Unified Option Handling Flow

User requested a cleaner approach:
1. Find scoreOption by looking up value
2. Check if scoreOption is NA option
3. If NA: check if has score â†’ skip or continue with NotApplicable=true
4. If regular: process as normal option with NotApplicable=false

### New Pattern for All Outcomes

```go
// Step 1: Find the value
var mappedValue *int
// ... determine mappedValue based on outcome/trigger type ...

// Step 2-4: Check option type and handle
if mappedValue != nil {
    isNA := isNumericValueNAOption(float32(*mappedValue), criterion)
    if isNA {
        // N/A option - check if it has a score configured
        if !doesNAOptionHaveScore(float32(*mappedValue), criterion) {
            continue // Skip creating score for N/A option without score
        }
        mappedScore.NotApplicable = true
    } else {
        // Regular option
        mappedScore.NotApplicable = false
    }
    mappedScore.NumericValue = nilOrFloat32(mappedValue)
}
```

### Applied to All Cases

1. âś… DETECTED (with/without Options)
2. âś… NOT_DETECTED (with/without Options)
3. âś… NOT_APPLICABLE
4. âś… StringValue (metadata trigger)
5. âś… BooleanValue (metadata trigger)
6. âś… NumberValue (metadata trigger)

### Benefits

- **Clearer logic:** Find value â†’ check NA â†’ handle NA vs regular
- **Single responsibility:** Each step does one thing
- **Consistent pattern:** Same flow for all outcome types
- **Easier to maintain:** Changes only touch one part
