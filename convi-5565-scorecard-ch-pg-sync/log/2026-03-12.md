## Investigation: ch_not_submitted Scorecards

### Setup

- **Customer:** Spirit (us-east-1-prod)
- **Date range:** 2026-02-01 to 2026-03-12
- **Method:** Export PG submitted scorecard IDs, cross-reference with CH `scorecard_d` table (distributed), identify those with `scorecard_submit_time = 0`
- **Result:** 26 scorecards (vs 31 from March 11 check — date range slightly different)

### Root Cause: UpdateScorecard async work overwrites SubmitScorecard async work

For all 26 scorecards, CH has exactly **1 row** per scorecard — from UpdateScorecard's async work (no submit_time). SubmitScorecard's async work either:
- Wrote with an earlier `update_time` and was merged away by ReplacingMergeTree, OR
- Failed silently

**How `update_time` works (confirmed in code):**
- `update_time = time.Now()` at the moment of the CH write (`conversation.go:2972`)
- `scorecard_last_update_time = scorecard.UpdatedAt` from the PG re-read (`conversation.go:2966`)
- `scorecard_submit_time = scorecard.SubmittedAt` from the PG re-read (zero if not yet committed)
- `scorecardAsyncWorkReadFromDB` re-reads PG at the **start** of async work (`action_create_scorecard.go:789`)

**Timeline for each affected scorecard:**
1. Auto-scoring creates unsubmitted scorecard on conversation close (writes PG + CH directly, no APIs involved)
2. Human opens scorecard from **Closed Conversations** UI
3. Human updates scores → **UpdateScorecard API** → PG commit → async goroutine starts
4. Human submits (quickly after update) → **SubmitScorecard API** → PG commit → async goroutine starts
5. Update's async re-reads PG **before** Submit's transaction commits → gets stale data (no submit_time)
6. Update's async processes + writes to CH with `update_time = time.Now()` (this is AFTER PG submit committed, but the data was read before)
7. Submit's async re-reads PG → gets submitted data → writes to CH with `update_time = time.Now()`
8. If Update's async finishes **after** Submit's, its `time.Now()` is higher → ReplacingMergeTree keeps the stale row

Evidence:
- **CH `update_time` is always AFTER PG `submitted_at`** (gap +0.4s to +282.7s, avg 83.7s) — the CH write happened after PG committed the submit, but the **data** was read before commit
- **CH `scorecard_last_update_time` is always BEFORE PG `updated_at`** (gap -0.3s to -93.8s) — confirms the async work read a stale PG version (before submit committed)
- **Only 1 row per scorecard in CH** — SubmitScorecard's write was either never made or already merged away by ReplacingMergeTree

### Submission Source Correction

Initially misidentified `submission_source=1` as "AUTO_SCORE". Code review revealed:

| Value | Enum | Meaning |
|-------|------|---------|
| 0 | `SCORECARD_SUBMISSION_SOURCE_UNSPECIFIED` | Not specified |
| 1 | `SCORECARD_SUBMISSION_SOURCE_CLOSED_CONVERSATIONS` | Submitted from Closed Conversations UI |
| 2 | `SCORECARD_SUBMISSION_SOURCE_QM_TASK_HOME` | Submitted from QM Task Home |

Source: `go-servers/apiserver/sql-schema/gen/go/protos/qa/qa.pb.go:131-134`

The auto-scoring code (`TriggerConversationAutoScoring` in `action_trigger_conversation_autoscoring.go`) does **not** call UpdateScorecard/SubmitScorecard APIs. It writes directly to PG via `scoring.CreateScorecardAndScoresInDB` and to CH via `clickhouse.WriteScores`/`clickhouse.WriteScorecards` — all synchronous, no async race. It also creates **unsubmitted** scorecards (no `SubmittedAt` set in `autoqa.MapToScorecard`).

### Why CLOSED_CONVERSATIONS Dominates

The typical flow: auto-scoring creates an unsubmitted scorecard on conversation close → human QA reviewer opens it from the "Closed Conversations" page → reviews/adjusts scores (UpdateScorecard) → submits (SubmitScorecard). If the user clicks "Save" then "Submit" in quick succession, both API calls' async goroutines race against each other for the CH write. This is the same race condition documented in the investigation, triggered by normal human UI interaction rather than programmatic back-to-back calls.

### Conclusion

The 26 ch_not_submitted scorecards are all caused by the documented async race condition (UpdateScorecard and SubmitScorecard called in quick succession from the Closed Conversations UI). This is the acceptable edge case — 26/2,996 = 0.87%, within the expected ~1% range. The `CLOSED_CONVERSATIONS` dominance (92%) reflects that this is the primary UI flow for scorecard review at Spirit.

No action needed — this is working as documented.
