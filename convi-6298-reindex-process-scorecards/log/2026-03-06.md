# Project Log - 2026-03-06

## Progress

- Simplifying PR #25916 based on backfill experience: focus on process scorecards ONLY
- Identified performance issues from Oportun backfill that also affect the Go workflow

## Details

### Issues identified from backfill experience

1. **Missing customer/profile in queries**: `scorecardQuery` does `WHERE str.type = ? AND s.created_at >= ? AND s.created_at < ?` without customer/profile. All `director.scorecards` indexes are prefixed with `(customer, profile)` — this would cause full table scans on large DBs like Oportun.

2. **JOIN on `scorecard_template_revisions` for type filtering**: The backfill showed this JOIN can cause issues (composite PK on template_revisions includes revision). Better to pre-fetch process template IDs with a direct query on `scorecard_templates WHERE type = 2`, then use `template_id IN (...)`.

3. **N+1 template revision fetches**: `buildHistoricScoresFromDirector` calls `fetchTemplateRevision` once per scorecard. With 43K scorecards but only 42 unique template revisions (Oportun), this is wasteful. Should cache by (template_id, revision).

4. **Over-generalized scorecard_types/template_ids**: The generalized loop over multiple types + `resolveTypesFromTemplateIDs` adds complexity for a feature we don't need yet. Simplify to process-only.

### Simplification plan

- Remove `scorecardTypes` loop — hardcode PROCESS (type=2)
- Extract customer/profile from job context, add to all queries
- Pre-fetch process template IDs from `scorecard_templates WHERE type = 2`
- Use `template_id IN (...)` instead of JOIN with `scorecard_template_revisions`
- Cache template revisions in memory (key: template_id + revision)
- Keep `scorecard_template_ids` as optional filter (useful for targeted backfills)
