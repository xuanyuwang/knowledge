### Key findings for multi-agent extension

- Prompt templates are self-contained per action type — new team actions can be added independently
- Conversation filter works on transcript lists — can handle cross-agent conversation pools

## Task Recommendation Exploration

Analyzed all 6 backend data sources to determine what coaching task recommendations they can support. Key findings:

**Strong data support (Priority 1):**
- Coach on declining criterion (from `RetrieveCoachingProgresses` score_delta)
- Review failed conversations (from `RetrieveQAConversations`)
- Create coaching plan (from `ListCoachingPlans` — agents without plans)

**Good data support (Priority 2):**
- Follow up on coaching opportunity (from `SuggestCoachingOpportunities`)
- Review ineffective plan (plans + progresses combined)
- Recognize improvement (positive score_delta)

**Cross-agent (Priority 3):**
- Team-wide weakness detection
- Agent priority ranking

**Key finding:** Most APIs already support batching via `agent_user_names` list, so fetching for multiple agents is efficient — single calls for progresses, plans, and opportunities.

See [task-recommendation-exploration.md](./task-recommendation-exploration.md) for full analysis.

## Prototype Implementation

Implemented the "Identify Agents Needing Coaching" feature — a prototype that takes a list of agents, identifies which ones need coaching and why, and ranks them by priority.

### Architecture

```
TeamCoachingIdentifier (orchestrator, holds gRPC stubs)
  → TeamCoachingDataFetcher (batch fetch, 4 API calls total)
  → TeamCoachingScorer (rule-based ranking, pure logic)
  → TeamCoachingModels (dataclasses)
```

### Files Created (8 files)

**Service modules (in `services/`):**
1. `team_coaching_models.py` — `CoachingPriority` enum, `CriterionAssessment`, `AgentCoachingAssessment`, `TeamCoachingResult` dataclasses
2. `team_coaching_data_fetcher.py` — Protocol-based callable DI, `TeamCoachingBackendClients`, `AgentRawData`, batch fetch + group by agent
3. `team_coaching_scorer.py` — Pure scoring logic: gap-from-target × 30 + declining × 2 + no-plan bonus 1.5 + opportunities × 0.5; priority thresholds: CRITICAL ≥5, HIGH ≥3, MEDIUM ≥1
4. `team_coaching_identifier.py` — Orchestrator with 4 gRPC batch wrappers, wires into `TeamCoachingBackendClients`

**Test files (in `tests/`):**
5. `test_team_coaching_scorer.py` — 9 tests covering all scoring rules
6. `test_team_coaching_data_fetcher.py` — 4 tests for batch grouping, missing clients, errors, empty input
7. `test_team_coaching_identifier.py` — 2 end-to-end integration tests

**Staging script (project root):**
8. `test_team_coaching_staging.py` — Standalone script to run against staging with real agent data

## Staging Validation & Iteration

Ran against voice-staging with 5 real agents from `customers/cresta/profiles/walter-dev`.

### Issues Found & Fixed

1. **Usecase format**: DB stores `walter-dev` but `ListCoachingPlans`/`SuggestCoachingOpportunities` reject plain usecase strings with `"Usecase name '<nil>' is not valid"`. `RetrieveCoachingProgresses` returns Internal Error with usecase. Fix: make usecase filter optional — all 3 APIs work fine without it.

2. **Sentinel scores**: All staging agents have `-1.0` scores (no QA data). This caused false "200% below target" alerts. Fix: added `NO_DATA_SENTINEL = -1.0` filtering in scorer — skip criteria with sentinel scores.

3. **Target field name**: Target proto uses `criterion_or_chapter_id` not `criterion_id`. Fix: updated `_build_target_map()`.

4. **Missing display names**: Staging criteria have empty `criterion_display_name`. Fix: fall back to `criterion_id`.

### New Features Added

1. **JSON data cache** (`services/team_coaching_cache.py`): Saves raw API responses and scored results to `.cache/team_coaching/` as human-readable JSON files for debugging. Files: `fetch_YYYYMMDD_HHMMSS.json` and `result_YYYYMMDD_HHMMSS.json`.

2. **REST endpoint** (`team_coaching_server.py`): Bottle-based REST server — no proto changes needed.
   ```

   GET  /health
   ```

## Endpoint Redesign: Vibe-Code Pattern

Explored PR #8990 (`coaching-plan-summary` app) in `chat-ai` repo. Key findings:

- The app **does not call CrestaAssistantService** — it calls Cresta backend gRPC services (CoachingService, AnalyticsService, etc.) directly from Flask
- Auth via `context.py` template: token passed from admin console → cookies → gRPC metadata
- Deployed to K8s `vibe-coded` namespace on `voice-staging` via `build_and_push.sh`
- Uses `crestapy.grpc.utils.create_sync_channel` with domain `grpc.{cluster_id}.cresta.ai:443`

**Decision**: Rewrite as a vibe-coded Flask app in `chat-ai` repo:
- Copy `scorer.py`, `models.py`, `cache.py` from python-ai-services (pure Python, no changes)
- Rewrite gRPC calls using `context.py` pattern (not env-var stubs)
- Deploy via vibe-code workflow

See [endpoint-redesign.md](./endpoint-redesign.md) for full architecture.

## Vibe-Coded App: `coaching-ai-summary`

Created the vibe-coded Flask app in `chat-ai` repo at `app/coaching-ai-summary/`.

### Two-Repo Relationship (Key Insight)

The handover doc shows the pattern clearly:
1. **chat-ai PR #8990** (Jan 27) = "Vibe code coaching plan summary app" — **prototype**, Flask app calling gRPC directly
2. **python-ai-services PRs** (Feb 23-26) = **production implementation** as proper gRPC service
3. **cresta-proto PRs** (Feb 17-24) = formal proto contract
4. **director PRs** (Feb 25-27) = frontend UI calling production service

For a hackathon, the vibe-coded app pattern is the right fit — move fast, no proto release needed.

### App Name

Chose `coaching-ai-summary` (not `team-coaching-identifier`) since we'll add more endpoints to this app.

### Files Created (8 files)

| File | Description |
|------|-------------|
| `app.py` | Flask app with `/api/identify`, `/api/users`, `/health` endpoints |
| `context.py` | Copied from vibe-coded template (token → cookies → gRPC metadata) |
| `models.py` | Pure Python dataclasses (copied from python-ai-services) |
| `scorer.py` | Pure scoring logic + `AgentRawData` (adapted from python-ai-services) |
| `cache.py` | JSON file cache for debugging (copied from python-ai-services) |

### Key Differences from python-ai-services Version

1. **No DI pattern** — gRPC calls are plain functions in `app.py`, not wrapped in Protocol classes
2. **Auth via context.py** — token from admin console → cookies → gRPC metadata (not env vars)
3. **`CustomerName`/`ProfileName` imports** — from `cresta.v1.customer.*_resourcename_pb2` (not `crestapy.resource_name`)
4. **`AgentRawData` moved to scorer.py** — uses plain lists instead of proto types for fields
5. **Python 3.13 required** — context.py uses `kw_only=True` dataclass parameter
