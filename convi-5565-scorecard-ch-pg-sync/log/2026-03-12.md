## Investigation: ch_not_submitted Scorecards

### Setup

- **Customer:** Spirit (us-east-1-prod)
- **Date range:** 2026-02-01 to 2026-03-12
- **Method:** Export PG submitted scorecard IDs, cross-reference with CH `scorecard_d` table (distributed), identify those with `scorecard_submit_time = 0`
- **Result:** 26 scorecards (vs 31 from March 11 check — date range slightly different)

### Root Cause: UpdateScorecard async work overwrites SubmitScorecard async work

For all 26 scorecards, CH has exactly **1 row** per scorecard — from UpdateScorecard's async work (no submit_time). SubmitScorecard's async work either:
- Wrote with an earlier `update_time` and was merged away by ReplacingMergeTree, OR
- Failed silently

Evidence:
- **CH `update_time` is always AFTER PG `submitted_at`** (gap +0.4s to +282.7s, avg 83.7s) — the CH write happened after PG committed the submit
- **CH `scorecard_last_update_time` is always BEFORE PG `updated_at`** (gap -0.3s to -93.8s) — the async work read a stale PG version (before submit committed)
- **Only 1 row per scorecard in CH** — SubmitScorecard's write was either never made or already merged away

### Key Patterns

| Pattern | Finding |
|---------|---------|
| **Submission source** | 24/26 (92%) are AUTO_SCORE, only 2 manual |
| **Score match** | 22/26 match, 4 have stale scores in CH |
| **Submitters** | 14 unique submitters — not user-specific |
| **Templates** | 3 templates (21 + 4 + 1) — not template-specific |
| **PG submit-update gap** | 25/26 have `submitted_at ≈ updated_at` (within 1ms) |
| **CH write delay** | CH `update_time` is 0.4-283s after PG submit (avg 84s) |

### Why AUTO_SCORE Dominates

AUTO_SCORE submissions call UpdateScorecard (to set AI scores) and SubmitScorecard back-to-back programmatically. The time between the two API calls is minimal (<100ms), triggering the async race condition. Manual submissions rarely trigger this because humans take seconds-to-minutes between updating scores and clicking submit.

### Conclusion

The 26 ch_not_submitted scorecards are all caused by the documented async race condition (UpdateScorecard + SubmitScorecard called within <100ms). This is the acceptable edge case documented in the investigation — 26/2,996 = 0.87%, within the expected ~1% range. The overwhelmingly AUTO_SCORE pattern (92%) confirms the mechanism: only programmatic back-to-back API calls trigger this at meaningful rates.

No action needed — this is working as documented.
