### Implementation Plan Summary

Four phases to implement `ext` tables:
1. Helper functions (`buildUsersExtTable`, `attachExtTablesToContext`)
2. Modify `parseClickhouseFilter` to return ext table + SQL referencing it
3. Thread ext table through to `QueryWithRetry` via context in all 19 callers
4. Update tests and golden SQL files

Key design decisions:
- Centralize in filter-building layer (not per call site)
- Feature flag `ENABLE_EXT_TABLE_FOR_USER_FILTER` (default false)
- `buildUsersExtTable` takes `[]string` (user IDs), not `[]*userpb.User`

### Performance Benchmark Results

ext table vs IN clause, measured on ClickHouse 24.2 with 10K seeded conversations:

| Filter Size | ext table | IN clause | Ratio |
|-------------|-----------|-----------|-------|
| 5 users | 4.3ms | 1.2ms | 3.5x slower |
| 50 users | 1.4ms | 1.3ms | ~equal |
| 500 users | 1.5ms | 2.6ms | 1.7x faster |
| 5,000 users | 2.4ms | 6.5ms | 2.7x faster |
| 10,000 users | 3.5ms | 11.6ms | 3.3x faster |

Decision: always use ext tables — worst-case overhead is ~3ms, negligible vs full query latency
