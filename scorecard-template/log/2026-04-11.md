## Details

### Key Insight 1: handleRemoveOption Already Handles N/A

The general `handleRemoveOption` function already handles the N/A case through its remapping logic (lines 155-159):

```tsx
const notApplicable = autoQANotApplicableField.value as number | null;
if (notApplicable != null) {
  const newNotApplicable = remapIndex(notApplicable);
  autoQANotApplicableField.onChange(newNotApplicable === -1 ? null : newNotApplicable);
}
```

When the deleted option **is** the N/A option, `remapIndex(notApplicable)` returns `-1`, which sets `auto_qa.not_applicable` to `null`. This means we don't need a separate `removeNAOption` function - the general deletion logic already handles it correctly.

### Key Insight 2: Feature Flag Already Cleaned Up

The `enableDuplicateScoreForCriteria` feature flag was already removed from main branch on April 9, 2026 (commit c4e5df3298 by Kurt Choi). The cleanup removed:
- The ternary checking the flag for normal options
- The legacy mode branch using `settings.options[index].value`
- The `resetAutoQADetectedFields` callback

My N/A feature branch still had this code because it was created before the cleanup. Removing it brings the branch in sync with main and removes 32 lines of dead legacy mode code.

**Result:** The codebase now assumes decoupled scoring mode everywhere. Legacy mode (where `option.value` was both the wiring ID and the score) is completely gone.
